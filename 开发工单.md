## 一、项目概述

### 1.1 项目背景

这是一个**自我进化的 AI 系统实验项目**。9 个 Hermes Agent 实例分成两队，形成一个闭环生态系统：

- **A 队（开发队）**：4 个 Agent，负责编写代码、创建 SKILL、管理记忆
- **B 队（质量队）**：4 个 Agent，负责代码审查、写测试、检查文档和安全
- **协调者（Agent 0）**：调度两队的产出、审核决策、执行 Git 提交

目标是 **零人工干预**——系统自己决定做什么、自己写代码、自己改进、自己提交到 Git。

所有 Agent 运行在 **本地免费模型** 上，不依赖任何商业 API。

### 1.2 核心能力

| 能力 | 说明 | 实现方式 |
|------|------|----------|
| 自我编码 | Agent 能创建和修改 .py 文件，实现功能 | delegate_task + dev-cell 技能 |
| 自我记忆 | Agent 能从过往错误中学习，不重复踩坑 | memory 工具 + 经验积累 |
| 自我审查 | 代码产出后有同伴审查，保证质量 | delegate_task + qa-cell 技能 |
| 自我进化 | 每轮循环系统都变得更智能、代码更健壮 | 30 分钟一轮，自动迭代 |
| 自我提交 | 所有改进自动 commit + push 到 GitHub | 协调者执行 git 命令 |
| SKILL 自建 | 发现可复用的模式自动创建 SKILL | skill_manage 工具 |

### 1.3 技术栈

| 层次 | 方案 | 选型理由 |
|------|------|----------|
| Agent 框架 | Hermes Agent CLI | 用户已有环境，支持 delegate_task |
| LLM 模型 | 本地开源模型（Ollama / llama.cpp GGUF） | 完全免费，数据不出本机，适合 24h 自动运行 |
| 调度方式 | cronjob（Hermes 内建） | 无需额外配置 |
| 子任务派发 | delegate_task | 原生支持，上下文隔离 |
| 技能系统 | SKILL.md（Hermes 原生） | 技能即记忆，可复用 |
| 代码仓库 | GitHub | 版本控制 + 可见进化轨迹 |
| Python 环境 | WSL 系统 Python 3 | 项目轻量，不需要额外依赖 |

**本地模型选择（按推荐优先级）：**

| 模型 | 运行方式 | 硬件要求 | 推荐场景 |
|------|----------|----------|----------|
| Qwen2.5-7B-Instruct (GGUF) | Ollama | 8GB VRAM / 16GB RAM | 主 Agent 推理（高理解力） |
| Qwen2.5-Coder-7B (GGUF) | Ollama | 8GB VRAM / 16GB RAM | 代码生成专用 |
| DeepSeek-Coder-V2-Lite (GGUF) | llama.cpp | 8GB VRAM / 16GB RAM | 备选 |
| Qwen2.5-3B-Instruct (GGUF) | Ollama | 4GB VRAM / 8GB RAM | 子 Agent 快速推理 |

**模型调用策略：**
- **协调者（Agent 0）**：用 7B 模型，需要较强的理解和决策能力
- **A 队（开发）**：统一用 Code 类模型（若可用）或统一模型
- **B 队（审查）**：用 7B 模型，不需要代码生成能力，但需要逻辑推理
- **所有子 Agent**：因为 Hermes Agent 的 `delegate_task` 默认共用用户当前配置的模型，若需更换模型需在 Hermes Agent 设置中配置 provider 切换

## 二、功能需求详述

### 2.1 调度与协调模块

#### 2.1.1 协调者（Agent 0）

**功能描述：** 每轮循环的主控者，负责启动循环、分发任务、审核结果、提交 Git。

**工作流程：**
1. 读取 TODO.md 了解当前待办任务
2. 读取 CHANGELOG.md 了解历史
3. 并行派发 4 个 A 队子 Agent（分别加载 dev-cell 技能）
4. 收集 A 队产物（代码文件、SKILL、memory 记录）
5. 并行派发 4 个 B 队子 Agent（分别加载 qa-cell 技能）
6. 汇总 B 队的审查报告
7. 决策：合并 / 驳回 / 需要修复
8. 执行 `git add` → `commit` → `push`
9. 更新 TODO.md 和 CHANGELOG.md

**关键技术细节：**
- 使用 `cronjob` 的 `workdir` 参数固定在项目目录
- 子 Agent 用 `delegate_task` 的 batch 模式并行
- 子 Agent 的输出通过 `tmp_agent/agent-{id}/` 目录透传
- 协调者本身不写代码，只做调度和决策

#### 2.1.2 cronjob 自动调度

**参数设计：**

| 参数 | 值 |
|------|-----|
| 调度频率 | `*/30 * * * *`（每 30 分钟） |
| 工作目录 | `/mnt/f/项目三：多Agent/` （WSL路径） |
| 加载技能 | `orchestrate-swarm` |
| 是否重复 | 是（无限循环） |
| 通知方式 | 不通知，静默运行 |

**第一轮手动触发验证：** 创建 cronjob 后手动 `run` 一次，检查：
- TODO.md 是否被读取和理解
- 子 Agent 是否成功创建和派发
- Git 是否完成提交和推送
- `tmp_agent/` 下是否有产出文件

### 2.2 开发队（A 队）模块

A 队由 4 个 Agent 组成，每个加载 `dev-cell` 技能，并行工作。

#### 2.2.1 Agent 1 — 核心逻辑（dev-core）

**职责：**
- 实现 TODO.md 中的核心功能
- 编写工具函数、算法逻辑
- 项目的主要 Python 文件编写者

**典型任务示例：**
- 实现 `swarm_tool.py` CLI 工具
- 实现 `agent_memory.py` 记忆读写工具
- 实现 `swarm_stats.py` 统计数据收集

**产出约束：**
- 每个函数必须有 4 层注释（作用/返回类型/设计理由/面试追问）
- 写完必须通过 Python 语法检查
- 写入 `tmp_agent/agent-1/output/`

#### 2.2.2 Agent 2 — 工具与接口（dev-tools）

**职责：**
- 开发项目工具函数
- 实现文件操作、数据格式化、日志等基础工具
- 确保其他 Agent 能调用这些工具

**典型任务：**
- 实现 `swarm_logger.py` 日志工具
- 实现 `file_utils.py` 文件操作封装
- 实现 `config_loader.py` 配置管理

#### 2.2.3 Agent 3 — 知识与技能（dev-skills）

**职责：**
- 创建新的 SKILL.md
- 更新已有 SKILL
- 编写 README、文档、注释

**典型任务：**
- 为每次发现的可复用模式创建 SKILL
- 更新 dev-cell 和 qa-cell 技能的版本
- 维护 CHANGELOG.md 和项目文档

#### 2.2.4 Agent 4 — 记忆与配置（dev-memory）

**职责：**
- 读取 TODO.md 分析优先级
- 记录经验到 memory（通过协调者间接完成）
- 管理和更新配置文件

**注意：** 子 Agent 不能直接调用 `memory` 工具，需要将需要记录的信息写入 report，由协调者代写。

### 2.3 质量队（B 队）模块

B 队由 4 个 Agent 组成，每个加载 `qa-cell` 技能，并行审查 A 队的产出。

#### 2.3.1 Agent 5 — 代码审查（qa-review）

**检查项：**
- 代码逻辑错误和边界情况
- Python 代码风格（PEP 8）
- 硬编码和魔法数字
- 异常处理是否完善
- 缺少类型注解

**严重级别：**

| 级别 | 含义 | 处理方式 |
|------|------|----------|
| critical | 代码有逻辑错误或语法错误 | 必须驳回 |
| major | 设计不合理但有解决方案 | 建议修复，协调者决策 |
| minor | 风格问题或可改进 | 记录，不阻塞合并 |
| suggestion | 未来优化方向 | 记录到 TODO |

#### 2.3.2 Agent 6 — 测试验证（qa-test）

**职责：**
- 为 A 队产出写单元测试
- 运行测试验证通过
- 检查现有测试是否仍然通过

**规范：**
- 测试文件命名 `test_*.py`
- 使用 `pytest` 运行
- 测试覆盖核心逻辑路径和边界情况

#### 2.3.3 Agent 7 — 文档审查（qa-docs）

**检查项：**
- 函数注释是否完整（4 层结构）
- README 是否需要更新
- 是否有 TODO/FIXME 遗留
- SKILL 描述是否准确
- CHANGELOG 是否记录变更

#### 2.3.4 Agent 8 — 安全与性能（qa-perf）

**检查项：**
- 潜在的安全问题（注入、路径遍历、命令执行）
- 性能瓶颈（不必要的循环、大内存分配）
- 资源泄漏（文件未关闭、连接未释放）
- 安全的默认值

### 2.5 进化循环设计

#### 每次循环的生命周期

```
时间线（~30 分钟）

T+0min   协调者启动，读取 TODO
T+1min   A队 4 个 Agent 并行启动（delegate_task batch）
T+1~8min A队各自开发：选任务→读上下文→写代码→验证→输出report
T+8min   协调者汇总A队产物
T+9min   B队 4 个 Agent 并行启动（delegate_task batch）
T+9~16min B队各自审查：读产物→审查→写测试→输出review
T+16min  协调者汇总审查报告
T+17min  决策：合并 | 驳回 | 修复
T+18min  执行 git add → commit → push
T+19min  更新 TODO.md + CHANGELOG.md
T+20min  本轮完成，等待下一轮
```

#### 进化方向（预期轨迹）

| 轮次 | 预期成果 |
|------|----------|
| Round 0 | 项目结构、SKILL、初始 commit（已完成） |
| Round 1 | 写出第一个工具函数、跑通完整调度 |
| Round 2-3 | 代码有测试覆盖、B队发现第一个 bug |
| Round 4-5 | Agent 从 memory 学到"上次测试失败因为 X" |
| Round 6+ | 自动重构重复代码、创建新 SKILL |
| Round 10+ | 系统自我诊断、性能分析、自动优化 |
| Round 20+ | 可能的涌现行为：Agent 之间协作模式进化 |

## 五、关键技术设计说明

### 5.1 本地免费 LLM 方案

**推荐方案：Ollama + Qwen2.5 系列**

Ollama 是最简单的一键运行本地模型的方案，支持 REST API（`http://localhost:11434`），Hermes Agent 可通过配置 `provider=ollama` 直接调用。

**安装 Ollama（WSL）：**
```bash
curl -fsSL https://ollama.com/install.sh | sh
```

**下载模型（RTX4060 8GB 推荐）：**
```bash
# 主 Agent 模型（7B，4bit 量化，~4.5GB VRAM）
ollama pull qwen2.5:7b-instruct-q4_K_M

# 如果需要 Coder 专用
ollama pull qwen2.5-coder:7b-instruct-q4_K_M
```

**Hermes Agent 配置（~/.hermes/config.yaml）：**
```yaml
providers:
  ollama:
    api_base: http://localhost:11434/v1
    model: qwen2.5:7b-instruct-q4_K_M
    api_key: ollama  # 本地模型不需要真实 key

# 如果8GB显存不够，可用3B模型跑子Agent
# 但 Hermes Agent 当前只支持单provider，所以所有 Agent（协调者和子Agent）共用同一个模型
```

**为什么不用商业 API：**
| 方案 | 成本 | 延迟 | 离线可用 | 隐私 |
|------|------|------|----------|------|
| Ollama 本地 | 0 元 | 中（8-15 tok/s） | ✅ | ✅ |
| DeepSeek API | 按量计费 | 快 | ❌ | ❌ |
| OpenAI API | 付费 | 快 | ❌ | ❌ |

**显存预算（RTX4060 8GB）：**
- Qwen2.5-7B Q4_K_M：~4.5GB → 剩余 3.5GB 给系统和 Hermes
- 若运行子 Agent 多实例，需注意显存上限。但 `delegate_task` 是顺序处理的，不会同时加载 4 个模型实例

### 5.2 cronjob 调度的核心机制

```python
# 调度器 cronjob 的核心 prompt（每次触发时执行）
"""
加载 orchestrate-swarm 技能。

执行一轮自我进化循环：
1. cd /mnt/f/项目三：多Agent/
2. 读取 TODO.md 和 CHANGELOG.md
3. 用 delegate_task 并行派发 A 队 4 个 Agent
   - 目标: [TODO 中未完成的任务]
   - 技能: dev-cell
   - 工具集: [terminal, file, skills]
4. 收集 A 队产物
5. 用 delegate_task 并行派发 B 队 4 个 Agent
   - 目标: [审查 A 队产出的文件]
   - 技能: qa-cell
   - 工具集: [terminal, file]
6. 汇总审查报告
7. 决策：合并改动 / 驳回 / 修复
8. git add -A && git commit -m "swarm-evolve: round N ..." && git push
9. 更新 TODO.md 和 CHANGELOG.md
10. 如果有需要记录的经验，写入 memory
"""
```

### 5.3 子 Agent 隔离机制

每个子 Agent 运行在完全隔离的上下文中：
- 不能访问 memory（防止污染）
- 不能调用 delegate_task（防止无限嵌套）
- 不能问用户问题（完全自动）
- 不能操作 Git（只有协调者能提交）

子 Agent 的 context 中必须包含：
```
工作目录: /mnt/f/项目三：多Agent/
项目根目录: /mnt/f/项目三：多Agent/
TODO.md 内容: [全文]
已经是第 N 轮循环
输出目录: tmp_agent/agent-{id}/
```

### 5.4 防死循环设计

| 风险场景 | 防护措施 |
|----------|----------|
| A 队产出同一个文件冲突 | A 队每个 Agent 分配固定文件范围，避免重叠 |
| B 队永远不通过 | 协调者有驳回和通过的决策权，多次驳回后强制合并 |
| 无限创建无用 SKILL | 只创建可复用的模式，每次创建需在前一轮 pr/issue 中说明 |
| Git 永远 no changes | 自动添加新 TODO 任务，确保总有待办 |
| cronjob 死锁 | 每轮 30 分钟，超时后自动下一轮 |

## 七、验收标准

| 验收项 | 通过标准 |
|--------|----------|
| 本地模型可用 | Ollama 运行中，Hermes 能通过本地 API 生成回复 |
| 协调者调度 | cronjob 触发后，协调者正确读取 TODO 并派发 A 队 |
| A 队产出代码 | 子 Agent 成功创建或修改 .py 文件，内容有意义，语法正确 |
| B 队审查 | 子 Agent 正确审查 A 队产出，输出 review.json |
| 审查反馈闭环 | 严重问题被驳回，小问题记录，无问题合并 |
| Git 自动化 | git commit + push 自动执行，commit message 有意义 |
| TODO 自动更新 | 已完成任务被打勾，有新任务时自动添加 |
| 连续运行 3 轮 | 没有人工干预的情况下，3 轮循环都正常完成 |
| memory 积累 | 至少有一次经验被写入 memory（由协调者代为写入） |
| 系统容错 | 子 Agent 故障时不阻塞整体循环，协调者有超时和 skip 逻辑 |

### 边界情况

| 场景 | 预期行为 |
|------|----------|
| Ollama 服务宕机 | cronjob 检测到 API 不可用后跳过本轮，等待下一轮 |
| TODO 为空 | 自动生成新任务（如"检查项目质量"、"添加新功能"） |
| 远程 Git 不可用 | 只做本地 commit，跳过 push，在 CHANGELOG 记录 |
| 子 Agent 全部失败 | 协调者跳过本轮，记录失败原因，下一轮继续 |
| 文件冲突（两个 Agent 改同一个文件） | 协调者检测到冲突，只保留一个版本，另一个驳回 |
| 连续 5 轮无变化 | 协调者主动生成"破局任务"（重构、优化、新功能） |
| 显存不足 | 考虑换用 3B 模型或减少并行数 |
