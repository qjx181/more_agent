# SWARM 运行规则（2026-05-20 更新）

## 核心定位 —— 持续项目优化引擎

**项目三是一个自进化的多 Agent 系统。它的使命不是「修Bug」，而是「持续优化任何项目」——覆盖代码质量、测试覆盖、性能、架构、代码风格、异步化、死代码清理、文档一致性、安全合规、依赖管理等所有维度。**

项目三的价值不限于找 Bug 和修 Bug。它能主动扫描项目代码库，发现一切可改进的地方，自动执行优化，验证效果，积累经验——然后下一次做得更快更好。

## 这跟「修 Bug」有什么区别？

| 旧定位（Bug 修复） | 新定位（持续优化） |
|---|---|
| 只在有 Bug 时有价值 | 每轮都有价值（优化是无限的） |
| 被动响应（等 Bug 出现） | 主动扫描（超前发现问题） |
| 关注错误（Traceback、异常） | 关注质量（测试覆盖率、代码异味、性能、架构） |
| 修好 Bug 就结束 | 持续提升项目基线（今天比昨天好） |
| 上限是「没有 Bug」 | 上限是「接近想得到的完美代码」 |

## 项目三能做什么优化（不限于 Bug）

- **代码质量**：死代码检测、未用import清理、深嵌套扁平化、硬编码值抽取
- **测试覆盖**：缺失的单元测试、边界情况、压力测试
- **异步化**：同步 I/O 阻塞检测、sync→async 转换
- **性能**：N+1 查询检测、不必要的循环、缓存策略
- **架构**：模块职责梳理、循环依赖检测、接口一致性
- **安全性**：注入检测、密钥泄露扫描、权限检查
- **文档**：README 过时、函数注释缺失、CHANGELOG 未更新
- **配置**：硬编码路径、魔法数字、环境变量抽取
- **依赖**：过时依赖、未使用的依赖

## 当前阶段

系统处于建设期——部分基础设施就位（code_review、bug_analysis、delegate_optimizer、parallel_dispatcher 等），但核心闭环（主动扫描→诊断→优化→验证→积累经验）尚未完全连通。

剩余待办：`self_cost_tracker_persistence`、`self_json_logs_startup_flag`、`self_metrics_sqlite_storage`

## 每轮工作流程

1. 扫描目标项目的全部维度（不止 Bug）→ 识别优化机会
2. 排序：按影响/成本比从高到低
3. 执行优化（协调者直接 write_file 或委托子 Agent）
4. 验证效果（语法检查、单元测试、基准对比）
5. Git 提交
6. 经验积累（哪种优化模式效果最好 → 下次优先采用）

## 注意事项

- 目标项目路径应通过 config.yaml 配置，不硬编码
- 优化"效果"的评判标准：代码质量提升 > token 消耗降低 > 纯文件数量变化
- 每轮结束后 self_evolve_log.json 中必须记录：本轮做了什么优化、效果如何、哪种方法值得复用
- 优化可以很小（删一行死代码），也可以很大（重写一个模块），关键是「每轮都有实际产出」
- 连续 3 轮没有找到优化机会 → 自动升级扫描力度（从浅层扫描升级到深层分析）
